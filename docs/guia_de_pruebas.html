★ El modelo en una frase

Cada torre = servidor de un sistema M/M/c (cola FIFO única); cada enemigo = cliente; una fuga = bloqueo por capacidad finita K. La temperatura (variable continua que decae exponencial hacia la ambiente —ley de enfriamiento de Newton—) apaga torres sobrecalentadas, reduce la capacidad efectiva y rompe el supuesto markoviano: por eso simular aporta sobre la fórmula cerrada. El objeto de estudio no es el juego, sino la validación del modelo de colas.

🔍 Recorrido del código SimPy — dónde y qué mirar

Toda la simulación vive en simulation.py. Estos son los 6 lugares clave: qué concepto de la materia implementa cada uno, el fragmento real y qué mirar. Las referencias son archivo : línea sobre la versión actual del repo.

Eventos discretos El reloj, la cola y los streams de azar simulation.py : 138–151
self.env   = simpy.Environment()            ← reloj + lista de eventos futuros (FEL)
self.queue = simpy.PriorityStore(...) if sc.priority else simpy.Store(...)  # cola FIFO o con prioridad
self.towers     = [Tower(i, sc) for i in range(sc.c)]   # c servidores
self.stream_arr = ExponentialStream(sc.seed)             # azar de arribos  (stream propio)
self.stream_srv = ExponentialStream(sc.seed + 10_007)    # azar de servicio (stream propio)

Qué mirar: simpy.Environment() ES el algoritmo de lista de eventos de la Unidad III: el reloj salta de evento a evento, no en pasos fijos. Store es la cola FIFO única; cambiarla por PriorityStore da la disciplina con prioridad. Un stream de azar por fuente = números aleatorios comunes (misma semilla ⇒ misma corrida, y comparaciones justas entre escenarios).

Llegadas Poisson + bloqueo ¿Cuándo aparece cada enemigo? simulation.py : 215–241
def arrivals(self):
    while True:
        yield self.env.timeout(self.stream_arr.sample(self._current_lam()))  ← Exp(λ): tiempo hasta el próximo arribo
        ...
        if self.sc.K > 0 and self._in_system() >= self.sc.K:   ← sistema lleno (capacidad K)
            self.res.leaked += 1                            # BLOQUEO = la "fuga" del juego
            self._ev(now, "leak", ...); continue
        self.queue.put(enemy)                            # si hay lugar, entra a la cola

Qué mirar: la línea del timeout(Exp(λ)) es el criterio de cuándo llega cada goblin: un proceso de Poisson de tasa λ (tiempos entre arribos exponenciales, sin memoria). El if … >= K es el sistema con pérdida: el cliente que llega con todo ocupado se pierde — esa es la fuga.

Servidor + servicio La torre atiende (FIFO, tiempo Exp(μ)) simulation.py : 244–271
def tower_proc(self, tower):
    while True:
        item = yield self.queue.get()      ← FIFO: toma el siguiente (espera si la cola está vacía)
        ...
        ts = self.stream_srv.sample(self.sc.mu * enemy["mu_factor"])  ← tiempo de servicio Exp(μ)
        yield self.env.timeout(ts)            # la torre "dispara" durante ts
        ...
        self._ev(now, "kill", ...)            # enemigo eliminado

Qué mirar: cada torre es un servidor; queue.get() implementa la disciplina FIFO (y espera sola cuando no hay trabajo); Exp(μ) es el tiempo de servicio. El mu_factor permite enemigos "fuertes/débiles" (V.A. discreta de tipo) sin tocar el motor.

Variable continua (EDO) La temperatura que "rompe la fórmula" simulation.py : 59–68 · 274–285
def temp_at(self, now):                  # variable CONTINUA de estado
    if self._heating: return self.temp + k_heat * dt          ← rampa lineal al disparar
    return Ta + (self.temp - Ta) * math.exp(-k_cool * dt)   ← decae exponencial hacia la ambiente (ley de Newton)
...
if tower.temp >= self.sc.T_max:          # se sobrecalentó
    tower.go_cooldown(now)                 ← la torre deja de atender: cae la capacidad efectiva
    yield self.env.timeout(cd)            # espera a enfriarse hasta T_resume

Qué mirar: la temperatura es la variable continua, integrada por una ley de enfriamiento exponencial (de Newton) en forma cerrada por tramos. Al entrar en cooldown la torre deja de ser un servidor disponible → rompe el supuesto markoviano (c servidores siempre activos) → la M/M/c queda como cota y por eso simular aporta.

Estimadores · Ley de Little De dónde salen Lq y L simulation.py : 167–182
def _integrate(self):        # estimación por ÁREA BAJO LA CURVA
    dt = now - self._last_int_t
    self.res.q_area   += len(self.queue.items) * dt   ← ∫ (largo de cola) dt  → Lq
    self.res.sys_area += self._in_system()     * dt   ← ∫ (clientes en sistema) dt → L

Qué mirar: L y Lq no salen de una fórmula, sino de integrar el estado en el tiempo (Lq = ∫cola dt / T) — el estimador estadístico de la simulación. Que después se cumpla Little (L = λ·W) es la prueba de consistencia interna (test sim_little, error ≈ 1.2%).

El motor Leer run() = leer el algoritmo de lista de eventos simulation.py : 304–309
def run(self):
    self.env.process(self.arrivals())              # entidad: llegadas
    for tw in self.towers: self.env.process(self.tower_proc(tw))  # entidades: torres
    self.env.process(self.monitor())               # foto del estado cada dt_sample
    self.env.run(until=self.sc.sim_time)          ← procesa la FEL en orden hasta el horizonte T

Qué mirar: registrás las entidades como procesos y env.run(until=T) extrae y procesa los eventos en orden cronológico hasta el horizonte. Eso es, literalmente, el esquema de lista de eventos futuros de la materia, sin escribir el while/heap a mano.

🔗 El azar que alimenta los timeout: los Exp(λ) / Exp(μ) se generan por transformada inversa en prng.py : 53 → exponencial_inversa(R, λ) = −(1/λ)·ln(1−R), a partir de un generador congruencial reproducible (prng.py : 22). Las fórmulas cerradas contra las que se valida (Erlang C / Erlang B) viven en analytical.py.
